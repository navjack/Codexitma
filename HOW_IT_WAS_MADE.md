# How Codexitma Was Made

This document explains how `Codexitma` was developed with Codex, how the project was steered over time, and which parts were primarily user-directed versus primarily generated by Codex.

This is a best-effort attribution document based on the development history in this repository's Codex sessions. It is meant to be honest and specific, not absolute. A lot of the final work is collaborative by nature: a user constraint, a Codex proposal, a user correction, and then a Codex implementation pass.

## Working Model

`Codexitma` was not produced from a single prompt or a single generated code dump.

The actual workflow was:

1. The user set product direction, constraints, and taste.
2. Codex proposed or implemented a concrete step inside those constraints.
3. The user tested the result in a live build.
4. The user then tightened the brief, corrected flaws, or expanded scope.
5. Codex refactored or extended the project to match the new target.

That loop repeated over a long span of development and is the main reason the project now has:

- a native macOS renderer
- an SDL cross-platform renderer
- a first-person `Depth 3D` mode
- a built-in adventure editor
- external JSON content packs
- automation hooks
- release packaging and CI

The user acted as product owner, creative director, tester, and final taste filter. Codex acted as the implementation engine, systems designer, and refactoring/packaging agent.

## What The User Clearly Drove

These were explicit user constraints, ideas, or repeated steering decisions that materially shaped the project:

- The Apple II / Ultima-inspired identity of the project.
- The insistence that the project should feel like `Codexitma` for a reason, not a generic retro RPG.
- The push away from a pure ANSI terminal game and toward real graphics.
- The requirement that graphics stay low-resolution and era-conformant rather than modern glossy pixel art.
- The demand for a larger, less linear RPG with more staying power: classes, character creator, traits, skills, replayability, equipment, stores, dungeons, NPCs, longer adventures.
- The request for a `Gemstone Warrior`-style graphics mode and later selectable visual themes.
- The requirement that graphical mode fit sensibly inside a normal desktop window, including 1080p constraints and later responsive UI expectations.
- The request for simple Apple II-style sounds.
- The push for externalized game data so adventures, items, NPCs, quests, shops, and related content live in data files instead of hardcoded Swift.
- The request for third-party adventures and mod-friendly pack loading.
- The insistence that the editor become a first-class mode rather than a launch-only switch.
- The push for Windows support and SDL parity.
- The request for a first-person `Depth 3D` mode with turn-in-place controls.
- The push for lighting, torches, shadows, ceilings, sky behavior, and debug visibility during depth-renderer iteration.
- The expectation that every working build be easy to access and that releases/artifacts be handled properly.
- The request for screenshot automation and ongoing documentation polish.

In short: the user repeatedly chose the product direction and raised the bar when the project became too narrow, too static, or too easy.

## What Codex Clearly Originated

These elements were primarily created by Codex rather than supplied directly by the user:

- The original `Ashes of Merrow` setting, naming, and initial adventure framing.
- Large parts of the concrete world-building, quest naming, item naming, and narrative texture inside the shipped adventures.
- The technical package architecture of the Swift project.
- The internal engine layering: engine/model/content/persistence/rendering/editor subsystems.
- The concrete JSON pack model and validation/export plumbing.
- The automation command grammar and screenshot-driving workflow.
- The native depth-lighting implementation details:
  - world-space light fields
  - shadow-mask caching
  - floor/ceiling shadow sampling
  - debug overlays
- The SDL frontend structure and parity work.
- The release automation and GitHub Actions build pipeline shape.
- The modularization/refactor work used to keep large files from becoming uneditable.

Codex also authored many implementation details that were never explicitly specified by the user but were necessary to make the requested features real.

## What Was Collaborative

A lot of the most important parts of `Codexitma` were not purely user-originated or purely Codex-originated. They were iterative.

### Visual Identity

The user chose the Apple II / Ultima direction and kept correcting the project back toward it. Codex translated that into:

- concrete palettes
- tile/sprite choices
- renderer behavior
- UI layout systems
- multiple visual theme implementations

### The Move To A Bigger RPG

The user identified that the game was too straightforward and not replayable enough. Codex responded by building:

- class-based hero templates
- traits and skills
- equipment and paper-doll support
- more content systems
- a second adventure

The user set the need; Codex supplied the system design and implementation.

### The Editor

The user pushed for a real graphical editor with usable authoring flow and external pack output. Codex designed and implemented:

- the document/editor model
- inspector/sidebar workflows
- save/export/playtest behavior
- pack templating and validation

The concept and pressure for usefulness came from the user. The actual editor architecture came from Codex.

### SDL / Windows Support

The user explicitly wanted SDL and Windows support and kept the project focused on practical parity rather than abstract portability. Codex handled:

- the SDL application structure
- parity fixes
- Windows CI lane setup
- artifact packaging
- screenshot parity workflows

### Depth 3D Lighting

The user pushed the renderer from a simple retro effect toward a more interesting lit world with torches, ceilings, sky behavior, and shadow debugging. Codex implemented the actual rendering approach, then revised it repeatedly based on user screenshots and feedback.

This is one of the clearest examples of genuine collaboration:

- user: taste, desired effect, critique, visual validation
- Codex: algorithm choice, code, iteration, debug tooling

## How The User Steered Development

The user did not mostly steer by specifying low-level code.

The user steered by:

- setting hard aesthetic constraints
- rejecting weak or overly simple versions
- insisting on specific missing capabilities
- testing live builds and reporting exact failures
- choosing which suggested next step to pursue
- pushing for extensibility and distribution, not just prototype completeness

That form of steering mattered more than line-by-line coding input. It is why the project kept growing in the right direction instead of stopping at a small demo.

## How Codex Added Value

Codex added value when the work required:

- translating vague goals into real architecture
- implementing large amounts of code quickly
- keeping multiple frontends in sync
- refactoring large files before they became too expensive to edit
- building release/CI infrastructure
- turning test feedback into concrete fixes
- creating supporting tools, scripts, and docs around the game itself

The strongest Codex contributions were breadth, iteration speed, and technical follow-through.

## Model Attribution

This project was developed across multiple Codex sessions and model revisions, including work attributed by the user to `Codex 5.3` and `GPT-5.4`.

The repository does not contain fine-grained, machine-verifiable provenance for each code block by exact model version. Because of that, the honest level of attribution is:

- some systems and early direction came from earlier Codex sessions
- later systems, refinements, refactors, and polishing passes were continued by newer GPT-5/Codex agents
- the final shipped state is the accumulated result of those sessions plus the user's continuous steering

If exact per-file or per-feature model provenance is needed later, the right way to do that would be to maintain a dedicated changelog or commit discipline that records model/session attribution at the time each feature lands.

## Short Version

If you want the shortest truthful summary:

- The user authored the direction, taste, constraints, testing, and major product decisions.
- Codex authored most of the implementation, systems architecture, refactors, tooling, and many of the original content details.
- The best parts of `Codexitma` are collaborative, because they were created through repeated user correction and Codex execution rather than either side working alone.
